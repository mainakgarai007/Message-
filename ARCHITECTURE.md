## Key Design Decisions

### 1. Per-Chat Bot Modes
**Decision**: Each DM and Group has independent bot mode  
**Why**: Allows fine-grained control - automation in some chats, manual in others  
**Impact**: More complex but much more flexible and powerful

### 2. Socket.io for Real-time
**Decision**: Use Socket.io instead of polling  
**Why**: True real-time updates, typing indicators, better UX  
**Impact**: More complex server setup but superior user experience

### 3. Language Auto-Detection
**Decision**: No user toggle, automatic detection  
**Why**: Seamless UX, users don't think about it  
**Impact**: Requires good regex patterns and fallbacks

### 4. MongoDB for Database
**Decision**: NoSQL MongoDB vs SQL  
**Why**: Flexible schema, good for chat apps, easy scaling  
**Impact**: Different query patterns, no joins

### 5. React Context API
**Decision**: Context API vs Redux  
**Why**: Simpler, built-in, sufficient for this app size  
**Impact**: Less boilerplate, easier to understand

### 6. JWT Authentication
**Decision**: JWT vs Session-based  
**Why**: Stateless, works well with REST APIs, scalable  
**Impact**: Token management in frontend, expiration handling

### 7. Email-Only Login
**Decision**: Email (no username) as unique ID  
**Why**: Simple, standard, easy verification  
**Impact**: Need email verification flow

### 8. No "Bot" in UI
**Decision**: Hide all automation indicators  
**Why**: Core requirement - must feel human  
**Impact**: Careful wording throughout UI

## Performance Considerations

### Frontend
- React.memo for expensive components
- Lazy loading for routes (future)
- Debouncing for search
- Pagination for message lists
- Local state management

### Backend
- MongoDB indexes on chatId, createdAt
- Pagination for all lists
- Efficient Socket.io room management
- Connection pooling
- Caching for About Me data

### Database
- Compound indexes for queries
- TTL indexes for expiring messages
- Proper schema design
- Regular backups

## Scalability Path

### Phase 1 (Current)
- Single server
- Single MongoDB instance
- PM2 cluster mode

### Phase 2 (Growth)
- Load balancer (Nginx)
- Multiple app servers
- MongoDB replica set
- Redis for sessions/caching

### Phase 3 (Scale)
- Kubernetes orchestration
- MongoDB sharding
- Separate Socket.io servers
- CDN for static files
- Message queue (RabbitMQ)

This architecture supports the core requirement: **A human-like messaging platform where automation is completely invisible.**
